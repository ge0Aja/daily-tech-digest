## Security & Privacy

### EchoLeak (CVE-2025-32711): Zero-Click Prompt Injection Silently Exfiltrates M365 Data
_Aim Labs / Varonis_ | [Read more](https://www.varonis.com/blog/echoleak)

EchoLeak (CVSS 9.3) is a zero-click prompt injection vulnerability in Microsoft 365 Copilot's RAG chatbot. A single crafted email triggers the AI to silently exfiltrate chat logs, OneDrive files, SharePoint content, and Teams messages with no user interaction. The exploit chains four bypasses: evading Microsoft's XPIA classifier, circumventing link redaction via Markdown reference links, exploiting auto-fetched images, and abusing a Teams-proxy CSP allowance. Microsoft patched it server-side without requiring client action, but the underlying RAG architecture design flaws may expose other LLM-integrated applications similarly.

### "Bleeding Llama" (CVE-2026-7482): 300,000 Ollama Servers Leaking Process Memory
_Cyera Research_ | [Read more](https://cybersecuritynews.com/ollama-vulnerability-exposes-servers/)

A critical heap out-of-bounds read in Ollama's GGUF tensor parsing pipeline (CVE-2026-7482) lets unauthenticated attackers upload a malformed GGUF file and leak the entire server process memory — including user prompts, conversation history, API keys, environment variables, and proprietary code. Approximately 300,000 internet-exposed Ollama instances are affected. Upgrade immediately to Ollama 0.17.1 and place an auth proxy or API gateway in front of all Ollama deployments — the framework was never designed for direct internet exposure.

### Apache HTTP/2 CVE-2026-23918: Critical Double-Free Enables DoS and RCE
_The Hacker News / SOCRadar_ | [Read more](https://socradar.io/blog/cve-2026-23918-apache-http-server-http-2/)

A double-free in Apache HTTP Server 2.4.66's mod_http2 module (CVSS 8.8) is triggered by a crafted HEADERS + RST_STREAM sequence, causing two nghttp2 callbacks to free the same stream memory. Researchers from Striga.ai demonstrated working RCE in lab conditions; a public proof-of-concept exists. Only version 2.4.66 is affected — upgrade to 2.4.67, which removes the vulnerable code path entirely. If immediate upgrade is not possible, temporarily disabling HTTP/2 (mod_http2) reduces exposure.
