### Exim "Dead.Letter" CVE-2026-45185: Unauthenticated RCE in BDAT/GnuTLS Builds, Found by Autonomous AI
_The Hacker News_ | [Read more](https://thehackernews.com/2026/05/new-exim-bdat-vulnerability-exposes.html)

CVE-2026-45185, nicknamed "Dead.Letter," is a critical unauthenticated RCE in Exim mail server versions 4.97–4.99.2 compiled with GnuTLS. The exploit triggers a use-after-free in BDAT message parsing: a TLS close_notify before body transfer completion, followed by one byte in cleartext on the same TCP connection, causes a write into freed heap memory enabling full code execution. Notably, the vulnerability was discovered by XBOW, an autonomous AI security research system — a signal of how AI-accelerated vulnerability discovery is reshaping the defender's timeline. OpenSSL-based Exim builds are unaffected; upgrade to Exim 4.99.3 immediately.

### CVE-2026-31431 ("Copy Fail"): Linux Kernel LPE Affects All Distros Running Kernels Since 2017
_Microsoft Security Blog_ | [Read more](https://www.microsoft.com/en-us/security/blog/2026/05/01/cve-2026-31431-copy-fail-vulnerability-enables-linux-root-privilege-escalation/)

CVE-2026-31431, dubbed "Copy Fail," is a high-severity (CVSS 7.8) local privilege escalation in the Linux kernel's AF_ALG (userspace crypto API) interface, abusing the interaction between AF_ALG sockets and the `splice()` system call to write into the kernel's page cache and corrupt privileged binaries like `/usr/bin/su` entirely in memory — making exploitation deterministic and undetectable via disk inspection. The flaw affects virtually all Linux distributions running kernels from 2017 onward (Red Hat, SUSE, Ubuntu, Debian, AWS Linux, and more), and is especially dangerous in multi-tenant Kubernetes clusters where it enables container breakout and lateral movement. Apply distribution kernel patches immediately.

### llm-d Joins CNCF Sandbox: Open Standard for Kubernetes-Native Distributed LLM Inference
_CNCF Blog_ | [Read more](https://www.cncf.io/blog/2026/03/24/welcome-llm-d-to-the-cncf-evolving-kubernetes-into-sota-ai-infrastructure/)

llm-d, an open-source Kubernetes-native distributed inference framework co-developed by IBM Red Hat, Google, CoreWeave, and NVIDIA, became a CNCF Sandbox project in March 2026 and is gaining rapid production adoption as the de facto open standard for large-model serving on Kubernetes. It disaggregates LLM inference into independently scalable prefill and decode phases across pods, and adds an LLM-aware routing layer that dispatches requests based on KV-cache state, pod load, and hardware characteristics. With backing from AMD, Hugging Face, Intel, Mistral AI, and multiple universities, llm-d provides a vendor-neutral alternative to proprietary inference stacks for teams deploying models in production.

## Systems & Engineering

### Kubernetes AI Conformance Program Defines New Standard for Training, Inference, and Agentic Workloads
_Google Open Source Blog_ | [Read more](https://opensource.googleblog.com/2026/04/kubernetes-goes-ai-first-unpacking-the-new-ai-conformance-program.html)

The CNCF's Kubernetes AI Conformance Program, introduced in April 2026, defines a standardized superset of Kubernetes conformance for AI/ML workloads across three categories: Training (distributed GPU jobs with gang scheduling), Inference (model serving with autoscaling and request routing), and Agentic (multi-step durable AI workflows). Dynamic Resource Allocation (DRA) is central to the standard, shifting resource assignment from coarse accelerator counts to fine-grained hardware attribute control. GKE and AKS are already certified; the program is led by engineers from Google, Microsoft, and Red Hat. For platform teams, this standard defines which Kubernetes distributions are fit-for-purpose for production AI.

### Linux Foundation Forms Working Group to Address Open Source Package Registry Sustainability Crisis
_ITOps Times_ | [Read more](https://itopstimes.com/devops/sonatype-joins-linux-foundation-initiative-to-address-open-source-sustainability-crisis/)

The Linux Foundation formed the Sustaining Package Registries Working Group in May 2026, uniting PyPI (Python Software Foundation), RubyGems (Ruby Central), Crates.io (Rust Foundation), Packagist, OpenVSX (Eclipse Foundation), and Sonatype under a shared effort to address a funding and security crisis in open source package infrastructure. AI-driven tooling has pushed registry downloads toward 10 trillion annually, triggering a sharp rise in bot traffic, automated publishing, security reporting volume, and registry abuse — exposing a sustainability gap that now constitutes a genuine software supply chain security risk. The group's objectives include developing viable funding models, coordinated security practices, and governance frameworks.
